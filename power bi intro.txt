In my recent cybersecurity project, the client was facing an issue to analyze threat information because the data was coming from multiple security applications and landing in various SQL Server tables. Each time the team wanted to perform analysis, they had to manually join multiple complex tables, which was extremely time-consuming.

Additionally, the client managed several customers, and analyzing customer billing, DevOps user stories, and Microsoft Sentinel logs required manually logging into each customer’s separate portal. This not only taking time but also created inconsistency in the analysis process.

To solve this, we built centralized Power BI reports that combined all the data in one place. This made threat analysis and customer reporting much faster and easier.

overview:

This project had two phases.

Phase 1 – Customer Information & Reporting

The client had multiple customers, and each customer had their own DevOps, Sentinel, and Log Analytics environments.
The Data Engineering team collected this data and loaded the daily Sentinel logs and DevOps details into Azure Blob Storage.

Using this data, I created Power BI reports to show:

How many user stories were created for each customer

Previous (backlog) user stories

Who created the user stories and when

The current status (Completed / Not Completed)

Similarly, I built Power BI visuals to display Microsoft Sentinel alert details and their status.

For customer cost and billing insights, the client provided M Query from Log Analytics.
Using that query, I created Power BI visuals to show:

Total resources used by each customer

Cost and billing amount per customer

Parameter-based filtering to compare customers easily

✅ Do you need to explain fields to the Data Engineering team?

Yes — but only the fields that are needed for reporting.

You don’t need to explain every field in the database.
You only explain:

what data you need

which columns are important

how those fields should be used in reports

any calculated logic or business rules

any missing fields that you need for Power BI
